# Техническое задание: Система регистрации на мероприятия с QR-кодами и интеграцией Яндекс.Форм

---

## 1. Общая концепция

Система позволяет организовать полный цикл регистрации участников на мероприятия:
1. **Сбор заявок** через внешнюю форму (Яндекс.Формы) или прямой ввод в админке.
2. **Автоматическая передача** данных в базу сайта через Webhook.
3. **Генерация персонального билета** с уникальным QR-кодом для каждого участника.
4. **Отправка билета** на email участника.
5. **Отметка о прибытии** (check-in) путём сканирования QR-кода на входе.
6. **Восстановление билета** с подтверждением через 6-значный код.
7. **Администрирование** через панель FilamentPHP.

## 3. Интеграция с Яндекс.Формами

### 3.1. Общий принцип работы

```
[Пользователь] → [Яндекс.Форма] → [Webhook] → [API Laravel] → [База данных]
```

Яндекс.Формы выступают **внешним фронтендом** для сбора заявок. Когда пользователь заполняет и отправляет форму, Яндекс делает POST-запрос на URL вашего сайта. Laravel принимает данные и сохраняет их в таблицу `participants`.

### 3.2. Настройка на стороне Яндекса

**Шаг 1. Создание формы**
- В конструкторе Яндекс.Форм создаётся форма с полями (Имя, Email, Телефон и любые другие).
- **Обязательно** добавляется **скрытое поле** `event_id` — оно будет заполняться автоматически из URL.

**Шаг 2. Настройка Webhook**
- Раздел «Интеграции» → «HTTP-запрос».
- URL: `https://events.elprof.ru/api/yandex/register`
- Метод: POST.
- Формат данных: JSON.
- Дополнительный заголовок (для безопасности): `X-Yandex-Webhook-Secret: ваш_секретный_ключ`.

**Шаг 3. Публикация формы**
- Ссылка на форму публикуется, вставляется в iframe на сайте с параметром `event_id`:
  ```
  https://forms.yandex.ru/u/XXXX/?event_id=1
  ```
- Параметр `event_id` автоматически попадает в скрытое поле формы.

### 3.3. Обработка на стороне Laravel

**Маршрут:** `POST /api/yandex/register` (освобождён от CSRF-проверки).

**Логика контроллера:**
1. Принять POST-запрос.
2. Проверить заголовок `X-Yandex-Webhook-Secret` (защита от спама).
3. Извлечь `event_id` из данных.
4. Проверить, что мероприятие с таким ID существует.
5. Извлечь имя, email, телефон из полей формы.
6. Проверить, не зарегистрирован ли уже такой email на это мероприятие (защита от дубликатов).
7. Создать запись в `participants` с автоматической генерацией `checkin_token`.
8. Сохранить **все** ответы в поле `answers` (JSON).
9. Вернуть `200 OK` Яндексу (иначе он будет повторять запросы).

**Что делать, если запрос не прошёл:**
- Яндекс повторяет запрос до 3 раз с интервалом.
- Логировать ошибки для ручного разбора.

## 4. Генерация и работа с QR-кодами

### 4.1. Токен для QR-кода
- При создании участника автоматически генерируется `checkin_token` — случайная строка из 40 символов.
- Токен используется в URL для билета и чек-ина.
- Токен невозможно подобрать (защита от несанкционированной отметки).

### 4.2. Что содержит QR-код
QR-код кодирует URL:
```
https://events.elprof.ru/checkin/{checkin_token}
```

### 4.3. Генерация изображения QR-кода
- Используется библиотека `bacon/bacon-qr-code` (уже установлена в проекте).
- QR-код генерируется в формате SVG (для чёткости на любых экранах).
- Генерация происходит «на лету» при просмотре билета.

## 5. Клиентские страницы (публичная часть)

### 5.1. Страница билета `/ticket/{token}`
- Доступна без авторизации.
- Показывает: название мероприятия, дату, имя участника, QR-код.
- QR-код можно сохранить или показать на входе.
- Кнопка «Скачать PDF» (сохранить билет).
- Кнопка «Добавить в календарь» (Google Calendar, Apple — генерация .ics).

### 5.2. Страница чек-ина `/checkin/{token}`
- Открывается при сканировании QR-кода.
- Проверяет, не был ли участник уже отмечен.
- Если нет — отмечает (`checked_in_at = now()`, `status = 'arrived'`).
- Показывает результат: «Гость Иван Петров успешно отмечен!» или «Уже был отмечен в 10:15».
- Доступна только авторизованным администраторам (или по IP офиса).

### 5.3. Страница восстановления билета `/recovery`
- Форма ввода email.
- После отправки — генерация 6-значного кода и отправка на почту.
- Редирект на страницу ввода кода.

### 5.4. Страница ввода кода `/recovery/code`
- Форма для ввода 6-значного кода.
- Если код верный и не истёк (15 минут) — отправка письма с ссылкой на билет.
- Очистка кода после успешного ввода.
- Защита от перебора: не более 5 попыток за 10 минут.

- Содержит: 6-значный код, инструкцию, срок действия.

### 6.3. Письмо-напоминание (опционально)
- Отправляется за 1 день до мероприятия.
- Содержит ссылку на билет и карту проезда.

## 7. Администрирование (FilamentPHP)

### 7.1. Ресурс `EventResource`
- Управление мероприятиями.
- Вкладка «Участники» (связанная таблица).
- Кнопка «Отправить билеты всем».

### 7.2. Ресурс `ParticipantResource`
- Таблица со всеми участниками.
- Фильтры: по мероприятию, статусу, дате регистрации.
- Колонки: имя, email, мероприятие, статус (бейдж), время отметки.
- Массовые действия:
  - «Отправить билеты выбранным».
  - «Отметить как прибывших».
  - «Экспорт в Excel».
- Одиночные действия:
  - «Отметить прибывшим» (вручную).
  - «Отправить билет повторно».
  - «Отменить регистрацию».

### 7.3. Виджеты на Dashboard
- Счётчик: всего регистраций / прибыло / ожидается.
- График регистраций по дням.
- Список ближайших мероприятий с процентом заполняемости.

## 8. Artisan-команды

### 8.1. `tickets:send {event_id?}`
- Отправляет билеты участникам, у которых ещё не отправлены (`ticket_sent_at` is null).
- Можно указать конкретное мероприятие или всем.
- Выводит прогресс в консоль.

### 8.2. `import:yandex-csv {file}`
- Импортирует участников из CSV-файла (выгрузка ответов Яндекс.Форм).
- Автоматически генерирует `checkin_token`.
- Пропускает дубликаты по email+event_id.

## 9. Безопасность

### 9.1. Защита Webhook
- Проверка секретного заголовка `X-Yandex-Webhook-Secret`.
- Логирование всех входящих запросов.

### 9.2. Защита чек-ина
- Только авторизованные админы могут отмечать прибытие.
- Альтернативно: проверка по IP (только из офиса).

### 9.3. Защита восстановления
- Rate-limit: 3 запроса кода за 10 минут с одного IP.
- Rate-limit: 5 попыток ввода кода за 10 минут.
- Код одноразовый и сгорает через 15 минут.

### 9.4. Защита от дубликатов
- Проверка уникальности email в рамках одного мероприятия.

## 10. Edge Cases (нестандартные ситуации)

| Ситуация | Решение |
| :--- | :--- |
| Пользователь зарегистрировался дважды | Проверка email+event_id, возврат «уже зарегистрирован» |
| Пользователь потерял билет | Восстановление через `/recovery` с PIN-кодом |
| Пользователь не получил письмо | Повторная отправка из админки или через восстановление |
| QR-код не сканируется | Возможность ручного ввода токена в админке |
| Пользователь пришёл без билета | Поиск по имени/email в админке и ручная отметка |
| Мероприятие отменено | Массовое обновление статуса + уведомление по email |
| Импорт дубликатов из CSV | Пропуск существующих записей (email+event_id) |
| Запрос от Яндекса не прошёл | Повтор до 3 раз, логирование ошибок |

## 11. Флоу (User Journey)

```
[Организатор]
    ├── Создаёт мероприятие в админке
    ├── Настраивает Яндекс.Форму
    ├── Публикует ссылку на форму
    ├── Видит список участников в админке
    ├── Запускает рассылку билетов
    └── На входе сканирует QR-коды

[Участник]
    ├── Заходит на сайт
    ├── Переходит по ссылке на Яндекс.Форму
    ├── Заполняет форму → данные уходят в базу
    ├── Получает email с билетом и QR-кодом
    ├── Приходит на мероприятие
    ├── Показывает QR-код на входе
    └── Если потерял билет → восстанавливает через email + PIN
```
