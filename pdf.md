Ниже представлено подробное техническое задание на создание PDF-файлов (согласий на обработку персональных данных) с использованием библиотеки MPDF в экосистеме Laravel + Filament. Акцент сделан на хранении шаблонов в базе данных с системой плейсхолдеров и на соблюдении архитектуры обезличенного хранения персональных данных.

# Техническое задание: Модуль генерации персонализированных PDF-согласий (MPDF)

## 1. Цель и область применения

Модуль предназначен для автоматической генерации персонализированных юридических документов (согласий на обработку персональных данных) для участников мероприятий. Документы формируются на основе шаблонов, хранящихся в базе данных, с динамической подстановкой персональных данных участников. Модуль интегрируется в административную панель Filament и использует библиотеку MPDF для конвертации HTML в PDF.

Ключевые требования:
- Текст шаблона и все его параметры управляются администратором через интерфейс Filament без участия разработчика.
- Персональные данные участника (ФИО, адрес, паспорт и т.д.) не хранятся на сервере, а временно извлекаются из Яндекс.Форм через API в момент генерации.
- Готовый PDF-файл сохраняется на сервере в защищённом хранилище, а ссылка на него фиксируется в обезличенной записи участника.
- Поддерживается как одиночная, так и массовая генерация (например, для всех участников мероприятия).

---

## 2. Хранение шаблонов документов

### 2.1. Модель `DocumentTemplate`
Для хранения шаблонов создаётся новая таблица `document_templates` со следующими полями:
- `id` (PK)
- `name` – название шаблона (например, «Согласие на обработку ПД (основное)»)
- `slug` – уникальный символьный код (для программного обращения)
- `content` – HTML-код шаблона с плейсхолдерами (подробнее в разделе 4)
- `variables` – JSON-массив доступных переменных с описанием (помогает администратору)
- `is_active` – boolean, активен ли шаблон
- `created_at`, `updated_at`

### 2.2. Формат контента
Поле `content` содержит HTML-разметку, включая стили (CSS) и плейсхолдеры вида `{{ variable_name }}`. Это позволяет администратору визуально редактировать шаблон в WYSIWYG-редакторе, а плейсхолдеры будут заменены на реальные данные при генерации.

---

## 3. Администрирование шаблонов в Filament

Создаётся ресурс `DocumentTemplateResource` со стандартным набором страниц (List, Create, Edit).

### 3.1. Форма редактирования
- Поле `name` – текстовое, обязательно.
- Поле `slug` – текстовое, только латиница и подчёркивание, уникальное, заполняется автоматически транслитерацией.
- Поле `content` – **RichEditor** (WYSIWYG) с поддержкой HTML. Рядом выводится подсказка со списком доступных плейсхолдеров (из поля `variables`).
- Поле `variables` – `KeyValue` или `Repeater`, где можно задать перечень допустимых переменных и их описание. Пример:
  - Ключ: `full_name` → Описание: «ФИО участника»
  - Ключ: `address` → Описание: «Адрес регистрации»
- Поле `is_active` – Toggle.

### 3.2. Валидация
- Проверка, что в `content` используются только разрешённые плейсхолдеры (соответствующие ключам из `variables`).
- Запрет на использование потенциально опасного JavaScript (санитизация HTML).

---

## 4. Синтаксис плейсхолдеров и доступные переменные

### 4.1. Формат плейсхолдеров
В поле `content` используются двойные фигурные скобки: `{{ variable_name }}`. Это интуитивно понятно и легко парсится. Администратор вставляет их в нужные места текста.

### 4.2. Стандартный набор переменных для согласия
Для модуля согласий на обработку ПД предопределены следующие переменные (добавляются автоматически при создании шаблона или администратором):
- `full_name` – ФИО участника
- `passport_series` – серия паспорта
- `passport_number` – номер паспорта
- `passport_issued_by` – кем выдан паспорт
- `passport_issue_date` – дата выдачи паспорта
- `registration_address` – адрес регистрации
- `phone` – телефон
- `email` – email
- `event_title` – название мероприятия
- `event_date` – дата мероприятия
- `current_date` – текущая дата (на момент генерации)
- `organization_name` – полное наименование организации-оператора
- `organization_inn` – ИНН организации

Все эти данные, кроме `event_*`, `current_date` и `organization_*`, являются персональными и хранятся в Яндекс.Форме. При генерации они будут временно извлечены.

## 5. Процесс генерации PDF

Генерация может запускаться администратором через Filament (индивидуально или массово) и выполняться в фоновом режиме (Job).

### 5.1. Индивидуальная генерация
- Администратор открывает карточку участника (`ParticipantResource`) или список участников, выбирает действие «Сгенерировать согласие».
- Система определяет, какой шаблон использовать (например, активный шаблон для типа «согласие» или выбранный администратором).
- Процесс:
  1. Получить `answer_id` участника из таблицы `participants`.
  2. Запросить через `YandexFormsService` данные ответа (ПД).
  3. Извлечь необходимые переменные из ответа Яндекса.
  4. Загрузить активный шаблон из `document_templates`.
  5. Подставить данные в плейсхолдеры с помощью простого поиска/замены или шаблонизатора Blade (рендеринг из строки).
  6. Сгенерировать HTML-строку.
  7. Передать HTML в MPDF, настроить колонтитулы, нумерацию страниц.
  8. Сохранить PDF в `storage/app/private/consents/{participant_id}_{timestamp}.pdf`.
  9. Обновить поле `consent_pdf_path` в записи участника (хранить относительный путь).
  10. Уведомить администратора об успехе или ошибке.

### 5.2. Массовая генерация
- Администратор в списке участников применяет Bulk Action «Сгенерировать согласия».
- Создаётся задание очереди (Job) `GenerateConsentsBatch`, которое принимает массив `participant_ids` и ID шаблона.
- Для каждого участника выполняется та же логика, что и при индивидуальной генерации.
- Важно: массовая генерация может быть ресурсоёмкой (обращения к API Яндекса). Необходимо предусмотреть задержки между запросами и обработку исключений для отдельных участников, не прерывая всего пакета.
- **Rate limiting:** не более 5 запросов к API Яндекс.Форм в секунду. При ошибке 429 — exponential backoff (1с → 2с → 4с), максимальная 3 попытки.
- **Отслеживание прогресса:** использовать `Illuminate\Bus\Batch` для массовой генерации. Администратор видит прогресс (X из Y) в реальном времени.
- По завершении администратор получает уведомление с количеством успешных/неудачных генераций.

### 5.3. Статусы генерации
В таблицу `participants` добавляются поля:
- `consent_status` – enum: `pending` | `generating` | `completed` | `failed`
- `consent_error` – текст ошибки (без ПД), nullable
- `consent_generated_at` – timestamp, nullable

В Filament статус отображается бейджем: pending (серый), generating (синий), completed (зелёный), failed (красный с тултипом ошибки).

### 5.4. Шаблон по умолчанию
При первом запуске (seed-migration) создаётся дефолтный шаблон «Согласие на обработку ПД» с готовым юридическим текстом по 152-ФЗ. Администратор может редактировать его или создавать новые шаблоны.

## 6. Обработка персональных данных при генерации

Следуя архитектуре обезличенного хранения, модуль **не сохраняет** персональные данные участника в своей базе или файлах логов.

- ПД (ФИО, адрес и пр.) извлекаются из Яндекс.Форм **только на время выполнения задачи** (в памяти процесса).
- После подстановки в HTML и создания PDF, переменные, содержащие ПД, уничтожаются.
- В логи записываются только технические идентификаторы (`participant_id`, `template_id`), успех/ошибка. Запрещено логировать значения плейсхолдеров.
- Сгенерированный PDF-файл содержит ПД, поэтому хранится в защищённой директории `storage/app/private`, недоступной из веба напрямую.

## 7. Структура PDF-документа

### 7.1. Формат страницы
- A4 (по умолчанию), поля: верхнее 20 мм, нижнее 20 мм, левое 25 мм, правое 15 мм.
- Шрифт: DejaVu Sans (или другой кириллический, загруженный в MPDF).
- Размер шрифта: 12pt, межстрочный интервал: 1.5.

### 7.2. Колонтитулы
- Верхний колонтитул: логотип организации (слева) и название документа (по центру).
- Нижний колонтитул: «Страница {PAGENO} из {nbpg}» (справа). MPDF автоматически подставляет эти значения, поэтому нам не нужно знать количество страниц заранее.

### 7.3. Стилизация
Все стили задаются внутри HTML-шаблона (inline или через `<style>`). MPDF поддерживает большинство CSS2.1 и некоторые свойства CSS3. Администратор может управлять внешним видом через редактор шаблона.

## 8. Хранение сгенерированных документов и доступ к ним

### 8.1. Файловая система
- PDF-файлы хранятся в `storage/app/private/consents/`.
- Имя файла: `{participant_id}_{timestamp}.pdf` или `{participant_id}.pdf` (перезаписывается при повторной генерации).
- Символическая ссылка на `private` диск не создаётся, прямой веб-доступ невозможен.

### 8.2. Доступ администратора
- В карточке участника (Filament) отображается кнопка «Скачать согласие», если `consent_pdf_path` не пуст.
- При нажатии вызывается метод контроллера/маршрута, который авторизует доступ (администратор), читает файл с диска и отдаёт его как `StreamedResponse` с соответствующими заголовками (`Content-Type: application/pdf`, `Content-Disposition: attachment`).
- URL для скачивания может быть одноразовым или подписанным (signed URL), чтобы избежать перебора.

## 9. Безопасность и целостность

- **Авторизация:** Все действия с шаблонами и генерацией доступны только аутентифицированным администраторам с соответствующими правами (можно через Filament Policies).
- **Проверка плейсхолдеров:** Перед сохранением шаблона проверяется, что все плейсхолдеры есть в списке разрешённых. При генерации отсутствующие переменные заменяются на пустую строку с предупреждением в лог.
- **Защита от инъекций:** HTML-контент шаблона санитизируется (удаляются скрипты, обработчики событий). MPDF не исполняет JavaScript, но для чистоты лучше очистить.
- **Целостность файла:** После генерации можно сохранять контрольную сумму (SHA256) файла в БД, чтобы позже проверять, не был ли файл подменён.
- **Срок хранения:** Предусмотреть механизм автоматического удаления старых PDF (например, через год после мероприятия) с очисткой поля `consent_pdf_path`.

### 8.3. Удаление старых документов
- Scheduled Command `CleanupOldConsents` запускается ежедневно.
- Логика: найти мероприятия с `end_date` < now() - 1 год, для каждого участника удалить PDF-файл и очистить `consent_pdf_path`.
- Удаление: физическое (`File::delete`) + обновление записи в БД.
- Логировать количество удалённых файлов.

## 10. Интеграция с MPDF

### 10.1. Конфигурация
- Пакет `carlos-meneses/laravel-mpdf` (или нативный `mpdf/mpdf`). Настройки MPDF (шрифты, размеры страницы, режим кеширования) выносятся в `config/mpdf.php`.
- Временная папка для MPDF должна быть доступна для записи (обычно `storage/tmp`).

### 10.2. Использование
- В сервис-классе (`PdfGeneratorService`) инкапсулируется создание экземпляра MPDF, передача HTML, добавление колонтитулов (через `SetHTMLHeader`/`SetHTMLFooter`), сохранение в файл.
- HTML предварительно рендерится через `Blade::render($template->content, $variables)` с limited context (только данные, без доступа к App/Facades). Это даёт поддержку `@if`, `@foreach` в шаблонах (полезно для условных блоков, например, показывать серию паспорта только для россиян). Строковая замена — как fallback.

### 10.3. Обработка ошибок
- Если MPDF не может обработать HTML (битые теги и т.п.), исключение перехватывается, логируется, администратору возвращается сообщение об ошибке.
- Для массовой генерации ошибка по одному участнику не должна останавливать весь процесс.

## 11. Дополнительные рекомендации

- **Предпросмотр PDF:** В форме редактирования шаблона добавляется кнопка «Предпросмотр» (Filament Action), которая открывает модальное окно с iframe. Генерирует PDF с тестовыми данными:
  - ФИО: Иванов Иван Иванович
  - Паспорт: 1234 567890, выдан ОВД г. Москвы, 01.01.2020
  - Адрес: г. Москва, ул. Тестовая, д. 1, кв. 1
  - Мероприятие: Тестовое мероприятие, 01.01.2025
  PDF генерируется на лету, не сохраняется на диск. При закрытии модалки — временный файл удаляется.
- **Версионирование шаблонов:** Для аудита изменений можно хранить историю редактирования (`document_template_versions`) с копией контента.
- **Расширяемость:** Система плейсхолдеров и шаблонов может использоваться и для других типов документов (сертификаты, приглашения) без изменения архитектуры.

Настоящее ТЗ описывает законченный модуль, который органично вписывается в существующую систему мероприятий, соблюдает требования по обезличенному хранению ПД и предоставляет администраторам полный контроль над содержанием и процессом генерации юридически значимых документов.

## 12. Аудит и логирование

### 12.1. Таблица аудита
Таблица `consent_generation_logs`:
- `id` (PK)
- `participant_id` (FK → participants, nullable)
- `template_id` (FK → document_templates)
- `status` – success | failed
- `error_message` – nullable, текст ошибки без ПД
- `generated_by` – ID администратора (FK → users, nullable)
- `created_at`

### 12.2. Правила логирования
- Записывать: `participant_id`, `template_id`, `status`, `error_message`, `generated_by`, timestamp.
- Запрещено логировать: ФИО, адреса, паспортные данные, содержимое PDF.
- Журнал доступен в Filament через отдельную страницу или в карточке участника.

## 13. Цифровая подпись (опционально, не в MVP)

- Реализация — по отдельному ТЗ после запуска MVP.

## 14. Загрузка шаблонов из .docx

Администратор загружает шаблоны в формате .docx через админку.

### 14.1. Процесс
1. Загрузка .docx через `FileUpload` в `DocumentTemplateResource`.
2. Конвертация .docx → HTML через `phpoffice/phpword`.
3. **Автоопределение плейсхолдеров:** система анализирует текст и автоматически заменяет известные поля на `{{ variable }}`.
4. Результат (HTML с плейсхолдерами) отображается в `RichEditor`.
5. Администратор проверяет и корректирует плейсхолдеры при необходимости.
6. Сохранение HTML в поле `content` шаблона.

### 14.2. Маппинг плейсхолдеров (автоопределение)
| Текст в .docx | Плейсхолдер | Описание |
|---------------|-------------|----------|
| Ф.И.О. / фамилия имя отчество | `{{ full_name }}` | ФИО участника |
| серия ______ номер | `{{ passport_series }}`, `{{ passport_number }}` | Паспорт |
| выдан ___________________ | `{{ passport_issued_by }}` | Кем выдан |
| по адресу: _______________ | `{{ registration_address }}` | Адрес регистрации |
| номер телефона | `{{ phone }}` | Телефон |
| адрес электронной почты | `{{ email }}` | Email |
| название мероприятия | `{{ event_title }}` | Мероприятие |
| «____» __________ 20__ г. | `{{ current_date }}` | Дата генерации |

### 14.3. Альтернатива
Администратор может создать шаблон без загрузки .docx — напрямую в `RichEditor`, используя .docx как образец для копирования текста.
