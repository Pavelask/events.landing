# Техническое задание: Система регистрации на мероприятия (обезличенное хранение)

---

## 1. Общая концепция

Система предназначена для организации регистрации участников на мероприятия с использованием собственного дизайна форм. Персональные данные (ФИО, email, телефон, ответы на кастомизированные вопросы) хранятся исключительно в Яндекс.Формах, а на сервере под управлением Laravel — только обезличенные технические идентификаторы и статусы. Связь между обезличенной записью и реальным человеком устанавливается через API Яндекс.Форм в реальном времени при необходимости.

**Ключевые решения:**
1. Новая система работает **параллельно** с существующей, не заменяя её. Создаётся отдельная модель `AnonParticipant` с отдельной таблицей `anon_participants`.
2. Маршруты `/ticket/`, `/checkin/`, `/recovery/` — **общие**. Контроллеры определяют тип участника (Participant или AnonParticipant) по `checkin_token` внутри логики.
3. OAuth access token задаётся в `.env` и обновляется вручную. Без автоматического refresh.
4. **API Яндекс Форм** требует заголовок `X-Cloud-Org-Id` с organization ID. Токен создаётся через OAuth приложение в Яндекс 360 для бизнеса с правами `forms:write` и `forms:read`.

## Этап 1: Инфраструктура и миграции

### 2.1. Таблица `form_templates` (шаблоны форм)

Назначение: хранение шаблонов кастомных полей форм, которые администратор создаёт через конструктор в Filament и привязывает к мероприятиям.

**Поля:**
- `id` — первичный ключ
- `name` — название шаблона (строка, обязательное)
- `questions` — JSON-массив объектов, описывающих дополнительные поля формы. Каждый объект содержит: `slug` (строка, уникальный в рамках шаблона), `label` (название вопроса), `type` (текстовый тип поля: text, textarea, select, radio, checkbox, date), `required` (булево значение), `options` (массив строк для select/radio/checkbox, пусто для text/textarea/date)
- `created_at`, `updated_at` — временные метки

### 2.2. Таблица `anon_participants` (обезличенные участники)

Назначение: хранение записей о регистрации участников **без персональных данных**. Только технические идентификаторы и статусы.

**Поля:**
- `id` — первичный ключ
- `event_id` — внешний ключ на таблицу `events` (обязательное)
- `answer_id` — строка, уникальный идентификатор ответа в Яндекс.Форме (уникальный индекс)
- `checkin_token` — строка длиной 40 символов, используется в URL билета для чек-ина (уникальный индекс)
- `status` — перечисление: `registered` (зарегистрирован), `arrived` (прибыл), `cancelled` (отменён). По умолчанию `registered`
- `checked_in_at` — временная метка чек-ина (nullable)
- `ticket_sent_at` — временная метка отправки билета (nullable)
- `souvenir_given` — булево, выдан ли сувенир (по умолчанию false)
- `documentation_given` — булево, выдана ли документация (по умолчанию false)
- `clothing_given` — булево, выдана ли одежда (по умолчанию false)
- `created_at`, `updated_at` — временные метки

**Индексы:** уникальный на `answer_id`, уникальный на `checkin_token`, обычный на `event_id`.

### 2.3. Таблица `newsletters` (история рассылок)

**Поля:**
- `id` — первичный ключ
- `subject` — тема письма (строка)
- `body` — тело письма (текст)
- `recipients_count` — количество получателей (целое, по умолчанию 0)
- `created_at`, `updated_at` — временные метки

### 2.4. Расширение таблицы `events`

Добавляется новое поле `form_template_id` — внешний ключ на таблицу `form_templates` (nullable,因为他не все мероприятия используют кастомные формы).

### 2.5. Переменные окружения

- `YANDEX_OAUTH_TOKEN` — access token из Яндекс OAuth для доступа к API Яндекс.Форм. Обновляется вручную.
- `YANDEX_ORG_ID` — ID организации Яндекс 360 (числовой ID владельца организации). Требуется для API Яндекс Форм.
- `YANDEX_FORM_ID` — идентификатор общей Яндекс.Формы, которая принимает ответы для всех мероприятий.

### 2.6. Конфигурация `config/services.php`

Добавляется секция `yandex` с параметрами `token` (из env `YANDEX_OAUTH_TOKEN`), `org_id` (из env `YANDEX_ORG_ID`) и `form_id` (из env `YANDEX_FORM_ID`).

## 3. Этап 2. Модели данных

### 3.1. Модель `FormTemplate`

- Заполняемые поля: `name`, `questions`
- Приведение типов: `questions` → массив (JSON)
- Связь: `events()` — один ко многим (один шаблон может быть привязан к нескольким мероприятиям)

- Методы `getStatusLabelAttribute()` и `getStatusColorAttribute()` — возвращают читаемое название и цвет статуса для отображения в интерфейсе

### 3.3. Обновление модели `Event`

Добавляются две новые связи:
- `formTemplate()` — многие к одному (мероприятие может иметь один шаблон формы)
- `anonParticipants()` — один ко многим (мероприятие может иметь nhiều анонимных участников)

## 4. Этап 3. Сервисы

### 4.1. Обновление `YandexFormsApi`

Существующий сервис уже содержит методы `createAnswer()`, `getAnswer()`, `getAnswers()`. 

**Критическое исправление: API Яндекс Форм требует organization ID:**

Все запросы к API должны включать заголовок `X-Cloud-Org-Id` с ID организации. Без этого заголовка API возвращает 401 Unauthorized или 400 "Требуется организация".

Методы должны быть обновлены:
- `createAnswer()` — добавить `X-Cloud-Org-Id` в заголовки
- `getAnswer()` — добавить `X-Cloud-Org-Id` в заголовки  
- `getAnswers()` — добавить `X-Cloud-Org-Id` в заголовки

Формат заголовков:
```php
Http::withHeaders([
    'Authorization' => 'OAuth ' . $this->token,
    'X-Cloud-Org-Id' => $this->orgId,
])
```

**Получение organization ID:**
- Organization ID можно получить из ответа `GET https://login.yandex.ru/info` — поле `id` или `psuid`
- Альтернативно: organization ID = numeric ID пользователя-владельца организации
- Organization ID нужно добавить в `.env` переменную `YANDEX_ORG_ID`

Дополнительно добавляется:
- Метод `findAnswersByEmail(string $email)` — получает все ответы формы и фильтрует их по полю email на стороне сервера. Используется при восстановлении билета и при проверке дубликатов.

### 4.2. Новый сервис `AntiBotService`

Комбинированная anti-bot защита, состоящая из трёх независимых методов. Все проверки проводятся **на сервере**. Если хотя бы один метод срабатывает — форма блокируется.

**Метод 1 — Honeypot (ловушка для ботов):**
- На странице формы размещается скрытое текстовое поле `website`, скрытое через CSS (позиционирование за пределами экрана, `aria-hidden="true"`, `tabindex="-1"`).
- Боты автоматически заполняют все поля формы, включая скрытые. Люди — нет.
- При отправке формы проверяется: если поле `website` непустое → блокируем отправку.

**Метод 2 — Time-check (проверка времени):**
- При загрузке формы (метод `mount`) в компонент записывается текущий Unix-timestamp.
- При отправке формы вычисляется разница между текущим временем и временем загрузки.
- Если прошло менее 3 секунд → блокируем отправку (пользователь не мог заполнить форму так быстро).

**Метод 3 — Math-question (математический вопрос):**
- При загрузке формы генерируются два случайных числа от 1 до 20.
- Пользователю показывается вопрос: "Сколько будет A + B?"
- Правильный ответ сохраняется в зашифрованном виде в сессию пользователя.
- При отправке формы сравнивается введённый ответ с правильным. Если не совпадает → блокируем.

## 5. Этап 4. Конструктор шаблонов форм в FilamentPHP

### 5.1. Ресурс `FormTemplateResource`

Размещается в административной панели Filament. Следует общему паттерну проекта: ресурс делегирует форму и таблицу отдельным классам.

**Список шаблонов (таблица):**
- Колонки: название, дата создания, дата обновления
- Навигация: группа "Настройки", иконка `heroicon-o-document-duplicate`

**Форма создания/редактирования:**
- Поле `name` — текстовое, обязательное
- Поле `questions` — повторитель (Repeater) с элементами, каждый из которых содержит:
  - `label` — название вопроса (текстовое, обязательное)
  - `type` — тип поля (выпадающий список: text, textarea, select, radio, checkbox, date)
  - `required` — переключатель (булево)
  - `options` — текстовая область для ввода вариантов ответа через запятую (отображается только для типов select, radio, checkbox)
  - `slug` — уникальный идентификатор вопроса (текстовое, автоматически генерируется транслитерацией из label, можно редактировать)

**Валидация:**
- `slug` — обязательное, уникальное в рамках шаблона, только латинские буквы и подчёркивание
- `options` — обязательное для типов select, radio, checkbox

### 5.2. Обновление `EventResource`

В секции «Регистрация» формы мероприятия добавляется:
- Поле выбора шаблона формы (`form_template_id`) — выпадающий список с relationship. При изменении выбранного шаблона (событие `afterStateUpdated`) выполняется копирование массива `questions` из шаблона в поле `questions` мероприятия. Это гарантирует, что при дальнейшем редактировании шаблона уже привязанные мероприятия не изменяются.
- Опция `yandex_api` в выпадающем списке `registration_type` — для обозначения мероприятий, использующих новую систему обезличенной регистрации.

## 6. Этап 5. Публичная страница и форма регистрации

### 6.1. Маршрут

- URL: `/events/{slug}/register-anon`
- Компонент: Livewire `AnonRegistration`
- Название маршрута: `event.register.anon`

### 6.2. Livewire-компонент `AnonRegistration`

**При загрузке (mount):**
- Загружается мероприятие по slug
- Проверяется, опубликовано ли мероприятие и открыта ли регистрация
- Генерируются случайные числа для математического вопроса
- Записывается timestamp загрузки формы

**Данные формы:**
- Базовые поля: Имя (текст, обязательно), Email (email, обязательно), Телефон (текст, необязательно)
- Динамические поля: рендерятся на основе `event.questions` (JSON из БД мероприятия)
- Скрытое поле `event_id`
- Поля anti-bot защиты: honeypot, math-answer, form-loaded-at

**Процесс отправки (submit):**
1. **Anti-bot проверка:** проверяются все три метода (honeypot, time-check, math-question). Если любой из них срабатывает — возвращается ошибка.
2. **Проверка дедлайна:** если у мероприятия задан `registration_deadline`, проверяется, что текущая дата не позже дедлайна.
3. **Проверка вместимости:** если у мероприятия задан `capacity`, подсчитывается количество участников в `anon_participants` и `participants` для данного мероприятия. Если сумма >= capacity — возвращается ошибка «Места закончились».
4. **Проверка дубликата:** через API Яндекс.Форм запрашиваются все ответы формы, фильтруются по email и event_id. Если найден дубликат — возвращается ошибка «Вы уже зарегистрированы».
5. **Формирование данных для Яндекс.Формы:** базовые поля (`event_id`, `name`, `email`, `phone`) плюс дополнительные поля. Маппинг: каждый вопрос из `event.questions` по порядковому номеру соответствует слоту `custom_1`, `custom_2`, ... `custom_N`.
6. **Отправка в Яндекс.Форму:** POST-запрос через `YandexFormsApi::createAnswer()` с общим `form_id` из конфигурации.
7. **Извлечение answer_id:** из ответа Яндекс.Формы извлекается идентификатор созданного ответа.
8. **Создание обезличенной записи:** в таблицу `anon_participants` записывается `event_id`, `answer_id`, сгенерированный `checkin_token` (40 символов), статус `registered`. Никаких персональных данных не сохраняется.
9. **Ответ пользователю:** сообщение об успешной регистрации. Билет не отправляется автоматически.

### 6.3. Blade-шаблон

- Динамический рендеринг полей формы на основе массива `questions`:
  - тип `text` → текстовое поле ввода
  - тип `textarea` → текстовая область
  - тип `select` → выпадающий список с опциями
  - тип `radio` → блок радио-кнопок с опциями
  - тип `checkbox` → блок чекбоксов с опциями
  - тип `date` → текстовое поле с маской ввода `ДД.ММ.ГГГГ` и плейсхолдером `01.01.1900`, валидация формата даты
- Учёт флага `required` для каждого поля
- Anti-bot элементы:
  - Скрытое поле honeypot (CSS: `position:absolute; left:-9999px`)
  - Видимый вопрос "Сколько будет A + B?" с числовым полем ввода
  - Скрытое поле с timestamp загрузки формы
- Стилизация через TailwindCSS v4, интерактивность через Alpine.js

## 7. Этап 6. Администрирование участников (FilamentPHP)

### 7.1. Ресурс `AnonParticipantResource`

Размещается в административной панели Filament. Состоит из ресурса, схемы формы, таблицы и страницы списка. Страницы создания и редактирования **не создаются** — участники попадают в систему только через публичную форму регистрации или импорт.

**Таблица (список участников):**

Колонки:
- ID
- Мероприятие (через связь `event`, вывод `title`)
- Статус (Badge с цветом: registered — серый, arrived — зелёный, cancelled — красный)
- Дата регистрации (`created_at`, формат дд.мм.гггг)
- Время чек-ина (`checked_in_at`, формат дд.мм.гггг чч:мм)
- Билет отправлен (иконка: есть/нет на основе `ticket_sent_at`)
- Отметки выдачи (иконки для сувенира, документации, одежды)

Фильтры:
- `event_id` — выбор мероприятия из выпадающего списка
- `status` — выбор статуса из выпадающего списка
- Наличие билета: отправлен / не отправлен (фильтр по `ticket_sent_at` null / not null)
- Диапазон дат регистрации (`created_at`)

Действия над одной записью:
- **«Просмотр карточки»** — открывает детальную страницу участника (см. п. 7.2)
- **«Отправить билет»** — по `answer_id` через API Яндекса получается email участника, отправляется письмо со ссылкой на билет, в БД обновляется `ticket_sent_at`
- **«Отменить регистрацию»** — меняет статус на `cancelled`
- **«Чек-ин»** — устанавливает `checked_in_at` текущее время, меняет статус на `arrived`

Массовые действия:
- **«Отправить билеты»** — для выбранных участников выполняется индивидуальная отправка с отчётом о количестве отправленных
- **«Экспорт обезличенный»** — скачивание CSV/Excel с полями из локальной БД (без ПД)
- **«Экспорт с ПД»** — запуск асинхронной задачи (см. Этап 7)

### 7.2. Карточка участника

Страница содержит три блока:

1. **Обезличенная информация** (из локальной БД): ID, мероприятие, статус, даты регистрации и чек-ина, отметки выдачи.

2. **Персональные данные** (загружаются временно):
   - Выполняется запрос к API Яндекса для получения ответа по `answer_id`
   - Извлекаются поля `name`, `email`, `phone`
   - Для каждого вопроса из `event.questions` определяется соответствующий слот `custom_*` (по порядковому номеру) и отображается его значение
   - **Никакие ПД не сохраняются на сервере** — данные отображаются только на время просмотра страницы

3. **Отметки о выдаче:** чекбоксы для сувенира, документации, одежды. При переключении сразу сохраняются в локальной БД (без ПД).

## 8. Этап 7. Экспорт с персональными данными (асинхронная задача)

### 8.1. Интерфейс

На странице списка участников администратор настраивает фильтры (мероприятие, статус и т.д.) и нажимает кнопку «Экспорт с ПД». Появляется предупреждение о содержании персональных данных и подтверждение.

### 8.2. Формирование задачи

Создаётся задание очереди (Job), которое получает параметры фильтрации и ID администратора. Задача выполняется асинхронно.

### 8.3. Логика выполнения задачи

1. Получение списка участников на основе фильтров (Eloquent-запрос к `anon_participants` с eager loading `event`).
2. Для каждого участника: запрос к API Яндекса для получения ответа, извлечение name, email, phone и значений `custom_*`.
3. Если затронуты мероприятия с разным набором вопросов — собираются все уникальные `slug` из всех затронутых мероприятий, формируется единая структура колонок. Для участников из мероприятий с меньшим набором вопросов пустые колонки заполняются пустыми значениями.
4. Генерация Excel-файла через `maatwebsite/excel`.
5. Сохранение файла во временное хранилище.
6. Отправка уведомления администратору через Filament Notification со ссылкой на скачивание. Ссылка действительна ограниченное время (1 час), после чего файл удаляется.

## 9. Этап 8. Импорт ответов из Яндекс.Формы

### 9.1. Интерфейс

В ресурсе `AnonParticipant` добавляется кнопка «Импорт из Яндекс Формы». Администратор указывает мероприятие, для которого импортируются ответы.

### 9.2. Процесс импорта

1. Запрос к API Яндекса для получения всех ответов общей формы с пагинацией (по 100 ответов за запрос).
2. Для каждого ответа:
   - Извлекается `answer_id` и значение поля `event_id`
   - Проверяется, что `event_id` совпадает с выбранным мероприятием
   - Проверяется наличие дубликата в таблице `anon_participants` по `answer_id` — если есть, пропуск
   - Проверяется существование мероприятия — если нет, пропуск с логированием
   - Генерируется `checkin_token`
   - Создаётся обезличенная запись в `anon_participants`
3. Вывод статистики: количество импортированных, пропущено (дубли), ошибок.

## 10. Этап 9. Билеты и восстановление (общие маршруты)

### 10.1. Просмотр билета

Маршрут `/ticket/{checkin_token}` — **общий** для обеих систем.

Логика контроллера:
1. Поиск участника: сначала в таблице `participants` по `checkin_token`, затем в `anon_participants`.
2. Определяется тип: `classic` (участник из `participants`) или `anon` (из `anon_participants`).
3. Если тип `anon`: загружается ПД из Яндекса по `answer_id` для отображения имени и email на билете.
4. Рендер шаблона билета с QR-кодом (ссылка на чек-ин).

### 10.2. Скачивание PDF-билета

Маршрут `/ticket/{checkin_token}/pdf` — аналогичная логика определения типа участника. Генерация PDF через DomPDF.

### 10.3. Восстановление билета

Маршрут `/recovery` — **общий**.

Логика:
1. Форма ввода email.
2. При отправке: поиск по email в обеих таблицах (`participants` и `anon_participants`).
3. Для `participants` — email хранится локально, поиск напрямую.
4. Для `anon_participants` — email не хранится, выполняется запрос к API Яндекса для поиска ответов с указанным email. Для каждого найденного ответа проверяется наличие записи в `anon_participants` по `answer_id`.
5. Собирается список всех найденных билетов (ссылки на билеты).
6. Отправляется одно письмо со списком всех билетов.
7. Rate limiting: 3 попытки с одного IP за 10 минут.

## 11. Этап 10. Чек-ин (общий маршрут)

### 11.1. Чек-ин по ссылке

Маршрут `/checkin/{checkin_token}` — **общий**. Доступен только авторизованным администраторам.

Логика:
1. Поиск участника: сначала в `participants`, затем в `anon_participants`.
2. Если запись найдена и `checked_in_at` пусто: устанавливается текущее время, статус меняется на `arrived`.
3. Если запись не найдена или уже отмечена — соответствующее сообщение.
4. Отображается результат (имя участника не показывается из соображений конфиденциальности).

### 11.2. Ручной чек-ин в админке

Реализован через action в ресурсе `AnonParticipantResource` — кнопка «Чек-ин» в таблице или карточке участника.

## 12. Этап 11. Безопасность

- **CSRF-защита** на всех формах (встроенная в Laravel).
- **Anti-bot защита** на публичной форме регистрации: комбинация honeypot + time-check + math-question. Полностью серверная проверка, без внешних зависимостей.
- **OAuth-токены** хранятся только в `.env`, никогда не передаются в браузер.
- **Логирование:** записываются только действия и технические идентификаторы (`answer_id`, `event_id`), без персональных данных.
- **Ссылки на скачивание** экспорт-файлов содержат случайный токен и действительны 1 час.
- **Rate limiting** на маршруте восстановления билета: 3 попытки с одного IP за 10 минут.
- **Все запросы к API Яндекса** выполняются по HTTPS.
- **Нехранение ПД на сервере:** персональные данные загружаются из Яндекса только временно (для просмотра карточки, отправки билета, формирования экспорта) и не сохраняются в БД сервера.
