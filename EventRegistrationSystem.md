# Техническое задание: Система регистрации на мероприятия с единой Яндекс.Формой и обезличенным хранением данных

## 1. Общая концепция
Система предназначена для организации регистрации участников на различные мероприятия с использованием собственного дизайна форм. Персональные данные (ФИО, email, телефон, ответы на кастомизированные вопросы) хранятся исключительно в Яндекс.Формах, а на сервере под управлением Rocky Linux — только обезличенные технические идентификаторы и статусы. Связь между обезличенной записью и реальным человеком устанавливается через API Яндекс.Форм в реальном времени при необходимости (просмотр карточки участника, экспорт с ПД, отправка билета).

Основные принципы:
- Используется **одна общая Яндекс.Форма** для всех мероприятий.
- Администратор через панель FilamentPHP создаёт шаблоны форм (конструктор), которые затем прикрепляются к конкретным мероприятиям.
- Публичная страница мероприятия рендерит кастомную форму на основе настроек мероприятия.
- При регистрации данные отправляются в общую Яндекс.Форму через API с указанием `event_id` и дополнительных полей, упакованных в предопределённые слоты `custom_1…custom_N`.
- На сервере сохраняется только обезличенная запись с уникальным `answer_id` и `checkin_token`.
- Все операции с персональными данными (отправка билетов, просмотр карточки, экспорт с ПД) выполняются путём временной подгрузки данных из Яндекса.

ВАЖНО! 
Реализовать отдельно от действующей системы регистрации, используя только данные действующих мероприятий!

## 2. Этап 1. Подготовка инфраструктуры и интеграция с Яндекс.Формами

### 2.1. Настройка OAuth-доступа
- Создаётся OAuth-приложение в Яндекс ID с типом «Веб-сервисы».
- Запрашиваются права: `forms:write` (для создания ответов) и `forms:read` (для чтения ответов).
- Получаются `client_id`, `client_secret`. Однократно вручную получается `refresh_token` и сохраняется в `.env` сервера.
- Реализуется механизм автоматического обновления краткосрочного `access_token` по `refresh_token` перед каждым обращением к API Яндекса.

### 2.2. Создание общей Яндекс.Формы
- В интерфейсе Яндекс.Форм создаётся одна форма, которая будет принимать ответы для всех мероприятий.
- Структура полей формы:
  - Скрытое поле `event_id` (обязательное).
  - Текстовое поле `name` (обязательное).
  - Поле email `email` (обязательное, с валидацией email).
  - Текстовое поле `phone` (необязательное).
  - Резервные текстовые поля `custom_1`, `custom_2`, …, `custom_N` (рекомендуется N=20). Они предназначены для передачи дополнительных вопросов.
- Идентификатор общей формы (`YANDEX_FORM_ID`) заносится в `.env`.

### 2.3. Определение маппинга дополнительных полей
- Для каждого мероприятия устанавливается соответствие: каждому дополнительному вопросу из конструктора назначается один из `custom_*` слотов.
- Маппинг может быть линейным: первый вопрос -> `custom_1`, второй -> `custom_2` и т.д., либо использовать `slug` вопроса для явной привязки.
- Порядок и соответствие сохраняются в JSON-поле `questions` мероприятия (каждый вопрос имеет поле `slug` и/или порядковый номер). При отправке данных в Яндекс.Форму система сопоставляет значения вопросов с соответствующими `custom_*` ключами.

## 3. Этап 2. Проектирование базы данных (обезличенные таблицы)

### 3.1. Таблица `form_templates` (шаблоны форм)
- `id` (PK)
- `name` – название шаблона
- `questions` (JSON) – массив объектов, описывающих дополнительные поля:
  - `slug` (строка, уникальный в рамках шаблона)
  - `label` (название вопроса)
  - `type` (enum: text, textarea, select, radio, checkbox)
  - `required` (boolean)
  - `options` (массив строк, только для select/radio/checkbox)
- `created_at`, `updated_at`

### 3.2. Таблица `events` (мероприятия)
- Стандартные поля: `id`, `title`, `slug`, `date`, `venue`, `capacity`, `registration_deadline` и т.д.
- `form_template_id` (внешний ключ к `form_templates`) – выбранный шаблон.
- `questions` (JSON) – копия вопросов из шаблона на момент привязки (или текущая версия). Гарантирует неизменность формы при дальнейшем редактировании шаблона.
- `yandex_form_id` – здесь может храниться общий идентификатор формы, либо можно брать из конфигурации.

### 3.3. Таблица `participants` (участники — только обезличенные данные)
- `id` (PK)
- `event_id` (FK к `events`)
- `answer_id` (строка, уникальная, идентификатор ответа в Яндекс.Форме)
- `checkin_token` (уникальная строка 40 символов, используется в URL билета)
- `status` (enum: `registered`, `arrived`, `cancelled`)
- `checked_in_at` (timestamp, nullable)
- `ticket_sent_at` (timestamp, nullable)
- `souvenir_given` (boolean, default false)
- `documentation_given` (boolean, default false)
- `clothing_given` (boolean, default false)
- `created_at`, `updated_at`
- Индексы: уникальные индексы на `answer_id` и `checkin_token`, индекс на `event_id`.

### 3.4. Таблица `newsletters` (история рассылок, опционально)
- `id`, `subject`, `body`, `recipients_count`, `created_at`.

## 4. Этап 3. Реализация конструктора форм в FilamentPHP

### 4.1. Ресурс `FormTemplateResource`
- Размещается в административной панели Filament.
- **Список шаблонов**: таблица с колонками `name`, `created_at`, `updated_at`.
- **Форма создания/редактирования**:
  - Поле `name` (обязательное).
  - Поле `questions` – `Repeater` со следующими подполями:
    - `label` (текстовое поле, обязательно).
    - `type` (выпадающий список: text, textarea, select, radio, checkbox).
    - `required` (переключатель).
    - `options` (текстовая область, отображается только для select/radio/checkbox; значения разделяются запятой).
    - `slug` (текстовое поле, автоматически генерируется транслитерацией из `label`, можно редактировать; должно быть уникальным в рамках шаблона).
- **Валидация**:
  - `slug` – обязательное, уникальное в шаблоне, только латиница и подчёркивание.
  - `options` – обязательное для типов select/radio/checkbox.

### 4.2. Ресурс `EventResource`
- В форму создания/редактирования мероприятия добавляется поле выбора шаблона формы (`form_template_id`, Select).
- При изменении выбранного шаблона (или при сохранении) выполняется копирование `questions` из шаблона в поле `questions` мероприятия. Это можно сделать через хуки `afterStateUpdated` у поля шаблона или в `beforeSave` мероприятия.
- Поле `questions` можно не показывать в форме, оно заполняется автоматически.

## 5. Этап 4. Публичная страница мероприятия и форма регистрации

### 5.1. Маршрут и контроллер
- URL: `/events/{slug}`.
- Загружается мероприятие с полем `questions` (JSON).
- Если мероприятие не опубликовано или регистрация закрыта, выводится соответствующее сообщение.

### 5.2. Динамический рендеринг формы
- Форма содержит базовые поля:
  - Имя (text, обязательно)
  - Email (email, обязательно)
  - Телефон (text, необязательно)
- Дополнительные поля рендерятся на основе `event.questions`:
  - Для каждого элемента массива создаётся поле соответствующего типа:
    - `text` → `<input type="text">`
    - `textarea` → `<textarea>`
    - `select` → `<select>` с опциями из `options`
    - `radio` → блок `<input type="radio">` с опциями
    - `checkbox` → блок `<input type="checkbox">` с опциями
  - Учитывается флаг `required`.
- Присутствует скрытое поле `event_id`.
- Интегрирована Google reCAPTCHA v3 (невидимая капча).

### 5.3. Отправка формы (контроллер или Livewire-действие)
1. Валидация: обязательные поля, email, капча.
2. Проверка дедлайна регистрации и наличия мест (запрос к локальной БД).
3. Проверка дубликата:
   - Используется API Яндекса: получение списка ответов общей формы с фильтром по email (или запрос всех ответов и фильтрация на стороне сервера).
   - Если найден ответ с тем же email и нужным `event_id`, возвращается ошибка «Вы уже зарегистрированы».
4. Формирование данных для создания ответа в Яндекс.Форме:
   - `event_id` = идентификатор мероприятия,
   - `name`, `email`, `phone`,
   - дополнительные поля: для каждого вопроса из `event.questions` значение кладётся в соответствующий `custom_*` согласно маппингу.
5. Отправка POST-запроса к API Яндекса (`/forms/{YANDEX_FORM_ID}/answers`) с авторизацией OAuth.
6. Из ответа Яндекса извлекается `answer_id`.
7. Генерация уникального `checkin_token` (40-символьная случайная строка).
8. Создание записи в таблице `participants`: `event_id`, `answer_id`, `checkin_token`, `status='registered'`. Персональные данные не сохраняются.
9. Возврат сообщения об успешной регистрации. Билет не отправляется.

## 6. Этап 5. Администрирование участников (FilamentPHP)

### 6.1. Ресурс `ParticipantResource`
- **Таблица (List)**
  - Колонки:
    - ID
    - Мероприятие (через отношение `event`, вывод `title`)
    - Статус (Badge с цветом: registered — синий, arrived — зелёный, cancelled — красный)
    - Дата регистрации (`created_at`)
    - Время чек-ина (`checked_in_at`)
    - Билет отправлен (Boolean иконка на основе `ticket_sent_at`)
    - Отметки выдачи (иконки для сувенира, документации, одежды)
  - Фильтры:
    - `event_id` (Select, список мероприятий)
    - `status` (Select)
    - Наличие билета: отправлен / не отправлен (фильтр по `ticket_sent_at` null/not null)
    - Диапазон дат регистрации (`created_at`)
    - Возможность сохранения предустановленных фильтров для быстрого доступа.

- **Действия над одной записью (Actions)**
  - **«Отправить билет»**:
    1. По `answer_id` через API Яндекса получается email участника.
    2. Отправляется письмо со ссылкой на `/ticket/{checkin_token}`.
    3. В БД обновляется `ticket_sent_at`.
  - **«Просмотр карточки»** – открывает отдельную страницу с детальной информацией (см. ниже).
  - **«Отменить регистрацию»** – меняет статус на `cancelled`.

- **Массовые действия (Bulk Actions)**
  - **«Отправить билеты»** – для выбранных участников выполняется индивидуальная отправка; прогресс и отчёт.
  - **«Экспорт обезличенный»** – мгновенное скачивание CSV/Excel с полями из локальной БД (ID, мероприятие, статус, даты, отметки).
  - **«Экспорт с персональными данными»** – запуск асинхронной задачи (см. раздел 7).

### 6.2. Карточка участника (ViewAction / EditAction)
- Страница содержит два основных блока:
  1. **Обезличенная информация** (из локальной БД): ID, мероприятие, статус, даты.
  2. **Персональные данные** (загружаются временно):
     - Выполняется запрос к API Яндекса: `GET /forms/{YANDEX_FORM_ID}/answers/{answer_id}`.
     - Из ответа извлекаются поля `name`, `email`, `phone`.
     - Для каждого вопроса из `event.questions` (мероприятия) определяется соответствующий слот `custom_*` (по порядковому номеру или slug) и отображается его значение.
  3. **Отметки о выдаче**: чекбоксы для сувенира, документации, одежды. При переключении сразу сохраняются в локальной БД (без ПД).

- Никакие ПД не сохраняются на сервере после завершения обработки запроса.

## 7. Этап 6. Экспорт с персональными данными (асинхронная задача)

### 7.1. Интерфейс
- На странице списка участников администратор настраивает фильтры (мероприятие, статус и т.д.) и нажимает кнопку **«Экспорт с ПД»** (доступна только авторизованным администраторам).
- Появляется предупреждение о том, что файл будет содержать персональные данные, и подтверждение.

### 7.2. Формирование задачи
- Создаётся задание очереди (Job), которое получает параметры фильтрации и ID администратора, запросившего экспорт.
- Задача выполняется асинхронно, чтобы не блокировать интерфейс.

### 7.3. Логика выполнения задачи
1. Получение списка участников на основе фильтров (через Eloquent).
2. Для каждого участника:
   - Запрос к API Яндекса для получения ответа (`GET /forms/{YANDEX_FORM_ID}/answers/{answer_id}`).
   - Извлечение ПД: `name`, `email`, `phone`, а также значений `custom_*`.
   - Формирование строки данных: ID, Мероприятие, Статус, Имя, Email, Телефон, и для каждого возможного дополнительного поля (из всех шаблонов или конкретного мероприятия) выводится значение.
3. Если мероприятий несколько с разными наборами полей, структура колонок унифицируется: собираются все уникальные `slug` из затронутых мероприятий, и для каждого участника значения подставляются только в соответствующие колонки (в остальных — пусто).
4. Генерация Excel-файла (например, с использованием `Laravel Excel`).
5. Файл сохраняется во временное хранилище (локальный диск или S3).
6. Администратору отправляется уведомление (через Filament Notification) со ссылкой на скачивание файла. Ссылка действительна ограниченное время (например, 1 час), после чего файл удаляется.

### 7.4. Безопасность и контроль доступа
- Файл доступен только аутентифицированному администратору, запросившему экспорт.
- Ссылка содержит случайный токен или подписывается.
- В логах фиксируется факт экспорта (какие фильтры, количество записей), без ПД.

## 8. Этап 7. Импорт ответов из Яндекс.Формы

### 8.1. Интерфейс
- В ресурсе `Participant` добавляется кнопка **«Импортировать ответы»**.
- Администратор может указать конкретный `form_id` (если используется не общая форма) или система берёт общий ID.

### 8.2. Процесс импорта
1. Запрос к API Яндекса для получения всех ответов формы с пагинацией (по 100 ответов за запрос).
2. Для каждого ответа:
   - Извлечение `answer_id` и `event_id` (из поля `event_id` ответа).
   - Проверка наличия в таблице `participants` записи с таким `answer_id`. Если есть – пропуск.
   - Проверка существования мероприятия с указанным `event_id`. Если нет – пропуск с логированием.
   - Генерация `checkin_token`.
   - Создание обезличенной записи в `participants`.
3. Вывод статистики: импортировано, пропущено (дубли), ошибок.

## 9. Этап 8. Отправка билетов (ручная) и восстановление

### 9.1. Индивидуальная и массовая отправка
- Реализовано в ресурсе `Participant` (Actions и Bulk Actions).
- Алгоритм уже описан в п. 6.1.

### 9.2. Страница восстановления билета
- `/recovery` – форма ввода email.
- При отправке система через API Яндекса ищет в общей форме все ответы с указанным email.
- Для каждого найденного ответа:
  - Извлекается `event_id` и проверяется, что мероприятие активно (или любое).
  - По `answer_id` находится запись в `participants` (если есть) и получается `checkin_token`.
  - Если билет существует, ссылка на него включается в письмо.
- Отправляется одно письмо со списком всех найденных билетов.
- Rate limiting: 3 попытки с одного IP за 10 минут.

## 10. Этап 9. Чек-ин (отметка о прибытии)

- Маршрут `/checkin/{checkin_token}` доступен только авторизованным администраторам (или по IP офиса).
- Логика:
  1. Поиск записи в `participants` по токену.
  2. Если `checked_in_at` пусто, устанавливается текущее время, статус меняется на `arrived`.
  3. Отображается сообщение об успешной отметке (без имени).
- Ручной чек-ин в админке (кнопка в карточке участника) выполняет то же самое.

---

## 11. Этап 10. Безопасность, логирование и анонимизация IP

- Все эндпоинты защищены CSRF-токенами.
- reCAPTCHA v3 на публичной форме регистрации.
- OAuth-токены хранятся в `.env`. Обновление `access_token` автоматизировано.
- Логи приложения: записываются только действия и технические идентификаторы (`answer_id`, `event_id`), без ПД.
- Логи веб-сервера: настроена анонимизация IP (замена последнего октета на 0) при ротации логов. Текущие логи для Fail2Ban сохраняют реальные IP до момента ротации.
- Все запросы к API Яндекса выполняются по HTTPS.

## 12. Заключение
Данный подход обеспечивает полное соответствие требованию о нехранении персональных данных на сервере, предоставляя при этом гибкий конструктор форм и удобное администрирование мероприятий. Фильтрация участников в админке позволяет быстро формировать выборки для рассылок и экспорта, а временная подгрузка ПД гарантирует конфиденциальность и минимальный след чувствительной информации в инфраструктуре заказчика.
